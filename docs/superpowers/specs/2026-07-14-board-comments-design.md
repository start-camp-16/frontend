# 뭐할구 프론트엔드 3단계: 게시판과 댓글 설계

## 목적

로그인 없이 태그별 게시글을 읽고 검색하며, 비밀번호를 요청 단위로 사용해 게시글과 댓글을 작성·수정·삭제하게 한다. 서버를 데이터의 단일 기준으로 삼고 실패 시 입력을 보존하되 비밀번호를 장기 보관하지 않는다.

## 의존성과 구조

1단계의 라우터, API, 모달, 페이지네이션, 비동기 상태 UI와 스타일을 사용한다. 게시글과 댓글은 같은 상세 화면에서 함께 동작하지만 API·뷰·폼 검증 모듈은 책임별로 나눈다.

```text
src/features/board/
  board-api.js
  board-constants.js
  board-list-page.js
  board-list-view.js
  post-detail-page.js
  post-detail-view.js
  post-form-page.js
  post-form-view.js
  post-validation.js
  comments-controller.js
  comments-view.js
  comment-validation.js
  board.css
```

## 라우트 책임

- `/posts`: 전체·태그 탭, 검색 폼, 게시글 목록, 페이지네이션
- `/posts/new`: 게시글 생성 폼
- `/posts/:id`: 게시글 상세와 게시글 삭제, 댓글 CRUD
- `/posts/:id/edit`: 게시글 조회 후 수정 폼

경로의 `id`는 1 이상의 정수만 허용한다. 유효하지 않은 ID나 `POST_NOT_FOUND`는 찾을 수 없음 상태와 게시판 복귀 링크를 표시한다.

## API 인터페이스

- `getPosts({ tag, q, page = 1, size = 20, signal })`
- `createPost({ tag, title, content, password }, { signal })`
- `getPost(postId, { signal })`
- `updatePost(postId, { password, tag, title, content }, { signal })`
- `deletePost(postId, { password }, { signal })`
- `getComments(postId, { signal }) -> Promise<Comment[]>`
- `createComment(postId, { content, password }, { signal })`
- `updateComment(commentId, { password, content }, { signal })`
- `deleteComment(commentId, { password }, { signal })`

각 함수는 `shared/openapi.yaml`의 메서드, 경로, JSON 필드와 상태 코드를 그대로 사용한다.

## 게시글 목록·검색

태그 순서는 `전체`, `관광`, `맛집`, `문화`, `행사`, `숙박`, `쇼핑`, `자유`로 고정한다. `전체`는 API의 `tag` 쿼리를 생략한다. 검색 폼의 현재 입력과 적용된 검색어를 구분하며 검색 버튼 또는 Enter 제출 시 공백을 제거한 값을 `q`에 반영하고 `page=1`로 이동한다. 빈 검색어 제출은 현재 태그 전체 목록으로 돌아간다.

URL의 `tag`, `q`, `page`가 목록 상태의 단일 기준이다. 지원하지 않는 태그는 `전체`, 유효하지 않은 페이지는 1로 정규화한다. 목록에는 태그, 제목, 작성일을 표시하고 제목은 상세 링크다. 응답 배열을 재정렬하지 않아 서버의 최신 작성순을 보존한다.

## 게시글 폼과 상세

생성과 수정 폼은 태그 1개, 제목 1~100자, 본문 1~5,000자, 비밀번호 4~20자를 제출 전에 검증한다. 길이는 앞뒤 공백을 제거한 제목·본문을 기준으로 하며 비밀번호는 공백을 포함한 입력값 그대로 길이를 검사한다. 첫 오류 필드로 포커스를 이동하고 필드별 메시지를 연결한다.

생성 성공은 응답의 `id`를 사용해 `/posts/:id`로 이동한다. 수정 화면은 상세를 먼저 로드해 태그·제목·본문만 채우고 비밀번호는 비워 둔다. 수정 성공도 상세로 이동한다. 서버 실패 시 태그·제목·본문과 현재 비밀번호 입력은 화면 재렌더링 없이 유지하되, 비밀번호를 모듈 상태·URL·저장소에 복사하지 않는다.

상세에는 태그, 제목, 본문, 생성 시각을 표시하며 `updated_at !== created_at`이면 수정 시각도 표시한다. 본문은 텍스트로 렌더링하고 줄바꿈만 CSS로 보존해 HTML 삽입을 막는다.

삭제 버튼은 확인 모달을 열고 비밀번호를 그 안에서 입력받는다. 성공 시 `/posts`로 이동한다. `PASSWORD_MISMATCH`는 비밀번호 필드 주변에 고정 문구 “비밀번호가 일치하지 않습니다.”로 표시하고 모달을 유지한다.

## 댓글

상세 진입 시 게시글과 댓글을 병렬로 요청하되 두 영역의 상태를 독립적으로 렌더링한다. 댓글은 API 순서를 그대로 사용한다. 댓글 생성 폼은 본문 1~1,000자와 비밀번호 4~20자를 검증한다. 성공하면 폼을 초기화하고 댓글 목록을 다시 가져온다.

각 댓글의 수정 버튼은 해당 항목 안에 본문·비밀번호 편집 폼을 열며 동시에 하나만 편집한다. 수정 성공 후 댓글 목록을 다시 요청한다. 삭제는 확인 모달에서 해당 댓글의 비밀번호를 입력받고, 성공 후 목록을 다시 요청한다. 모든 `PASSWORD_MISMATCH`는 작업 중인 폼/모달 안에서 안내한다.

댓글 새로고침에 실패해도 게시글 상세는 유지하고 댓글 영역에만 재시도를 제공한다. 댓글 작업 중에는 해당 버튼만 비활성화해 중복 요청을 막는다.

## 날짜, 오류, 접근성

유효한 ISO date-time은 한국어 로케일로 표시하고 원본 값은 `<time datetime>`에 둔다. 파싱 불가능한 값은 레이아웃을 깨지 않고 원문을 표시한다. 폼 전체 오류는 `aria-live`, 필드 오류는 `aria-describedby`로 연결한다.

삭제·수정 모달은 focus trap, Escape, 원래 버튼으로의 포커스 복귀를 지원한다. 탭은 버튼 역할과 선택 상태를 제공하고 좌우 방향키 이동도 지원한다. 요청 취소는 오류로 표시하지 않으며 서버 오류 메시지는 공통 보안 정책을 따른다.

## 테스트

- 태그 순서, 태그+검색어 결합, 빈 검색어 초기화, URL 뒤로 가기
- 목록 응답 순서 보존, 빈 목록, 오류 재시도, 페이지네이션
- 게시글과 댓글의 모든 최소·최대·경계 초과 검증
- 생성·수정 실패 시 입력 유지와 성공 후 라우팅
- 잘못된 비밀번호 메시지와 성공한 게시글 삭제 후 목록 이동
- 게시글 상세와 댓글 영역의 독립 로딩·오류
- 댓글 생성·수정·삭제 후 목록 재조회와 중복 제출 방지
- 본문을 HTML로 해석하지 않는 XSS 방어 회귀 테스트
- 모달과 태그 탭의 키보드·포커스 동작
- Playwright API mock으로 게시글 및 댓글 전체 CRUD와 비밀번호 불일치 흐름

## 완료 기준

- 모든 게시판 라우트에 직접 접근·새로고침할 수 있다.
- 태그, 검색어, 페이지와 브라우저 탐색 기록이 일치한다.
- 게시글과 댓글 CRUD가 OpenAPI 요청 형식으로 동작한다.
- 비밀번호가 요청 폼 밖의 상태, URL, 저장소에 남지 않는다.
- 오류가 발생한 영역만 복구 가능 상태로 바뀌며 다른 콘텐츠는 유지된다.
