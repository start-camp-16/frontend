# 뭐할구 프론트엔드 4단계: 챗봇과 출시 설계

## 목적

모든 화면에서 서울 장소와 커뮤니티 정보를 질문하고 근거가 포함된 답을 받을 수 있는 세션 기반 챗봇을 제공한다. 모바일 접근성, 공급자 장애 복구, 전체 기능 E2E와 Netlify 출시 조건까지 마무리한다.

## 의존성과 구조

1단계 앱 셸의 챗봇 마운트 지점, API 클라이언트, 모달·비동기 상태 UI를 사용하고 3단계 게시글 상세 라우트에 근거 링크를 연결한다. 장소 상세 라우트는 제품 범위에 없으므로 장소 근거는 링크가 아닌 구조화된 정보로 표시한다.

```text
src/features/chat/
  chat-api.js
  chat-controller.js
  chat-state.js
  chat-view.js
  chat-sources.js
  chat.css
tests/e2e/
  ranking.spec.js
  board-comments.spec.js
  chat.spec.js
  navigation.spec.js
```

## 공개 인터페이스

- `mountChat({ container }) -> () => void`: 앱 생명주기 동안 한 번 마운트하고 전역 플로팅 UI를 관리한다.
- `sendChat({ message, history }, { signal }) -> Promise<{ answer, sources }>`: `POST /api/chat`을 호출한다.
- `createChatState()`: `{ isOpen, messages, isSending, error }` 메모리 상태를 제공한다.
- `toChatHistory(messages) -> Array<{ role, content }>`: 전송 중인 메시지 이전의 성공한 user/assistant 메시지 중 최근 10개만 반환한다.
- `renderChatSources(sources)`: `type` discriminator로 장소와 게시글 근거를 안전하게 렌더링한다.

## 상호작용

플로팅 버튼은 모든 라우트에서 같은 위치에 유지되고 `aria-expanded`로 열림 상태를 알린다. 데스크톱에서는 우측 하단 패널, 360px 모바일에서는 화면 전체 오버레이로 연다. 닫기 후 플로팅 버튼으로 포커스를 복귀시키며 오버레이가 열린 동안 배경 포커스와 스크롤을 막는다.

메시지는 앞뒤 공백을 제거한 뒤 1~1,000자인지 검증한다. Enter는 전송하고 Shift+Enter는 줄바꿈한다. 요청 중 입력과 전송 버튼을 비활성화해 중복 전송을 막는다. 사용자 메시지를 먼저 화면에 추가하고 응답 성공 시 assistant 메시지와 근거를 추가한다.

대화 기록은 현재 페이지 탭의 JavaScript 메모리에만 존재한다. 라우트 이동에는 유지되지만 새로고침에는 사라진다. localStorage, sessionStorage, IndexedDB를 사용하지 않는다. API의 `history`에는 현재 전송 메시지를 중복하지 않고, 이전에 성공한 대화에서 최근 10개 메시지만 포함한다.

## 근거 표시

`location` 근거는 제목, 카테고리, 구, 선택적 주소를 표시한다. PRD와 API에 장소 상세 라우트가 없으므로 클릭 링크를 만들지 않는다. `post` 근거는 태그와 제목을 표시하고 `/posts/{post_id}` 링크를 제공한다. 알 수 없는 `type`은 전체 답변을 실패시키지 않고 해당 근거만 생략한다.

## 오류와 복구

- `CHAT_RATE_LIMITED`: “요청이 많습니다. 잠시 후 다시 시도해 주세요.”와 동일 메시지 재시도 제공
- `CHAT_PROVIDER_ERROR`: 외부 서비스 연결 실패를 알리고 동일 메시지 재시도 제공
- 그 밖의 오류: 안전한 서버 메시지 또는 일반 연결 오류와 재시도 제공
- 요청 취소: 오류 메시지를 추가하지 않음

실패한 사용자 메시지는 대화에 남기고 실패 상태와 재시도 버튼을 연결한다. 재시도는 같은 메시지를 다시 추가하지 않고 기존 항목의 상태만 갱신한다. 성공 전 실패 메시지는 이후 history에서 제외한다. 패널을 닫는 행위는 전송을 취소하지 않으며 앱 자체가 해제될 때만 요청을 취소한다.

## 출시 검증

Playwright는 실제 백엔드가 없어도 결정적으로 실행되도록 모든 API를 route mock한다. 1~3번 흐름은 1280×800 데스크톱에서, 4~6번 흐름은 1280×800과 360×800에서 각각 검증한다.

1. 직접 `/` 진입 → 메타데이터 → 랭킹 조회 → 페이지 이동 → 뒤로 가기
2. `/posts` 태그+검색 → 생성 → 상세 → 수정 → 잘못된 비밀번호 → 삭제
3. 댓글 생성 → 수정 → 잘못된 비밀번호 → 삭제
4. 모든 주요 라우트에서 챗봇 열기 → 답변과 장소/게시글 근거 → 라우트 이동 후 기록 유지
5. 챗봇 rate limit/provider 오류 → 재시도 성공
6. 새로고침 후 SPA 라우트 유지와 챗봇 메모리 초기화

프로덕션 빌드 후 미리보기 서버에서도 핵심 smoke test를 실행한다. 저장소 전체를 검색해 API 키 패턴, `localhost:8000`을 제외한 고정 서버 주소, 비밀번호 로깅이 없음을 확인한다. `VITE_API_BASE_URL` 설정 방법과 Netlify 환경변수·빌드·publish 설정을 README에 기록한다.

## 접근성·반응형 검증

- 챗봇 열기·닫기·메시지 작성·재시도를 키보드만으로 수행
- 오버레이 focus trap, Escape 닫기, 포커스 복귀
- 새 메시지와 오류를 과도한 중복 없이 `aria-live`로 알림
- 360px에서 입력 영역과 근거 카드가 잘리거나 가로 스크롤을 만들지 않음
- loading/error가 색상만이 아니라 텍스트와 상태 속성으로 구분됨

## 완료 기준

- 모든 라우트에서 챗봇을 열고 같은 탭 내 라우트 이동 후 대화를 이어갈 수 있다.
- OpenAPI 제한에 맞는 메시지와 최대 10개 history를 전송한다.
- 장소 근거는 정보로, 게시글 근거는 유효한 내부 링크로 표시된다.
- 특정 챗봇 오류가 명확한 한국어 안내와 동작 가능한 재시도를 제공한다.
- 단위·DOM 테스트, 전체 Playwright 핵심 흐름, 프로덕션 빌드가 통과한다.
- Netlify에서 직접 URL·새로고침과 환경변수 기반 API 통신이 동작한다.
- 모든 주요 화면에서 TourAPI·공공누리 출처 문구에 접근할 수 있다.
