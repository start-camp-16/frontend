# Project Working Agreement

## 기본 원칙

- 작업 전에 요청 범위와 현재 저장소 상태를 확인한다.
- 사용자의 기존 변경을 덮어쓰거나 관련 없는 변경을 커밋하지 않는다.
- 파괴적인 Git 명령을 사용하지 않는다.

## 작업 흐름

1. 구현 전에 `docs/superpowers` 아래에 작업 spec과 plan을 작성한다.
2. spec과 plan을 작성한 후 `feature/<작업명>` 브랜치를 생성한다.
3. 요청을 독립적으로 검증 가능한 기능 단위로 나누어 구현한다.
4. 각 기능 단위의 구현과 검증이 끝날 때마다 커밋한다.
5. 구현 완료 후 변경 범위와 diff를 검토하고 필수 테스트를 실행한다.
6. 기능 브랜치에 최신 `master`를 병합해 충돌 여부를 확인한다.
7. 충돌 해결 후 필수 테스트를 다시 실행한다.
8. 검증에 성공하면 `master`에 기능 브랜치를 병합한다.
9. 최종 상태를 확인한 후 `origin/master`에 push한다.

## 문서 관리

- `docs/superpowers`의 spec과 plan은 로컬 작업 참고 자료다.
- `docs/superpowers` 아래의 파일은 stage하거나 commit하지 않는다.

## 브랜치 규칙

- 브랜치명은 `feature/<작업명>` 형식으로 작성한다.
- `<작업명>`은 기능을 설명하는 영문 kebab-case를 사용한다.
- 예: `feature/ranking-location-api`

## 구현과 커밋

- 하나의 커밋에는 독립적으로 설명하고 검증할 수 있는 한 가지 기능 변경만 포함한다.
- 커밋 전에 변경 파일을 확인하고 관련 없는 파일과 `docs/superpowers` 파일을 제외한다.
- 각 기능 단위에 필요한 테스트를 실행한 뒤 커밋한다.

## 검토와 테스트

- 최소 검증으로 `npm test`와 `npm run build`를 실행한다.
- 사용자 흐름, 브라우저 동작 또는 통합 영역에 영향이 있으면 관련 `npm run test:e2e`도 실행한다.
- 기능 브랜치에 최신 `master`를 병합하거나 충돌을 해결한 뒤에는 최소 검증을 다시 실행한다.
- 테스트가 실패하면 병합하거나 push하지 않고 원인을 보고한다.

## 병합과 push

- 기능 브랜치에 최신 `master`를 먼저 병합한다.
- 충돌이 발생하면 변경 의도를 보존해 해결하고 diff와 테스트를 다시 확인한다.
- 충돌을 안전하게 해결할 수 없으면 작업을 중단하고 사용자에게 보고한다.
- 검증이 성공한 경우에만 `master`에 기능 브랜치를 병합한다.
- `master`의 최종 상태를 확인한 뒤 `origin/master`에 push한다.
